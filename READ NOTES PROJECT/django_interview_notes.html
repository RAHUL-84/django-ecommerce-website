Conceptual सवालों के जवाब

Q1. "Relation" कौन सा लगा है?

आपके models में मुख्यतः दो तरह के relations हो सकते हैं:

ForeignKey (One-to-Many)

जैसे Product का Category से रिश्ता:

class Product(models.Model):
    category = models.ForeignKey(Category, on_delete=models.CASCADE)

मतलब — एक Category में कई Products हो सकते हैं, पर एक Product सिर्फ एक Category का होगा।

लेकिन आपके Cart और OrderDetail model में phone/user को CharField जैसा इस्तेमाल किया गया है (ForeignKey नहीं), क्योंकि session में सिर्फ phone (text) स्टोर हो रहा है, Customer object नहीं। यहाँ कोई proper Django relation नहीं है — सिर्फ manual filtering हो रही है।
Interview में बोलना है: "मैंने Product-Category में ForeignKey relation इस्तेमाल किया, लेकिन Customer-Cart-Order को मैंने session-based phone number से manually link किया, प्रॉपर ForeignKey इस्तेमाल नहीं किया।"

Q2. URL कैसे बनता है (view → url → admin flow)

Browser Request→ urls.py→ views.py→ models.py→ template (HTML)→ Response 
# urls.py
urlpatterns = [
    path('search/', views.search, name='search'),
]

जब browser में http://127.0.0.1:8000/search/?query=EAR जाता है — Django urls.py में देखता है कि search/ किस function से जुड़ा है, फिर उस function को call करता है।

Django Admin अलग से auto-generate होता है — यह built-in feature है। बस admin.py में model register करना होता है: 
admin.site.register(Product, AdminProduct)
इसके बाद Django खुद-ब-खुद Add/Change/Delete UI बना देता है /admin/ पर।

Q3. makemigrations / migrate क्यों किया?

  • makemigrations — models.py में बदलाव चेक करके एक "blueprint" (migration file) बनाता है।
  • migrate — उस blueprint को असली Database में apply करता है, यानी table बनाता/बदलता है।
बिना migrate किए, database में table ही नहीं बनेगी, और Admin में उस model को register करने पर crash होगा (table not found)।
models.py→ makemigrations→ migrate→ admin.py register→ Admin Panel

Q4. base.html क्यों बनाई?

हर page में Navbar, Footer, CSS/JS, Bootstrap जैसी चीज़ें repeat होती हैं — बार-बार लिखना गलत practice है (DRY Principle)।

{% extends 'base.html' %}
{% block content %}
  <!-- सिर्फ उस page का unique content -->
{% endblock %}

href change क्यों होता है: navbar में hardcoded URL की जगह Django का {% url %} tag इस्तेमाल होता है:

<a href="{% url 'cart' %}">Cart</a>

फ़ायदा: urls.py में path बदलने पर भी {% url %} अपने-आप सही URL निकाल लेगा — manually हर file में href बदलना नहीं पड़ेगा। इसीलिए नया view/url बनाने के बाद base.html में href अपडेट करना पड़ता है।

भाग 2 — Project Problem 1

Add to Cart में सारे items एक साथ checkout हो रहे हैं

BUG

Flipkart जैसा behavior चाहिए — हर cart item का अपना अलग checkout/order हो, सबका मिलाकर एक साथ नहीं।

कारण: checkout view पूरे phone/session के सारे Cart items को एक साथ loop करके एक ही जगह से Order बना रहा है, individual item handle नहीं हो रहा।

Option A

हर Cart item का अलग "Buy Now" बटन हो, जो सिर्फ उस एक item का checkout खोले (cart_id पास करके)।

Option B

Checkout में सिर्फ उन items का Order बने जिन्हें ✅ checkbox से select किया गया हो (Flipkart जैसा)।

Exact fix के लिए checkout/order बनाने वाला view function चाहिए — भेजने पर exact लाइन बताई जाएगी।
भाग 3 — Project Problem 2

Search सिर्फ Home Page वाले Products दिखा रहा है

BUG

असली search function में query लिया तो गया, पर Product model को कभी query ही नहीं किया गया:

def search(request):
    totalitem = 0
    if request.session.has_key('phone'):
        phone = request.session["phone"]
        query = request.GET.get('query')
        totalitem = len(OrderDetail.objects.filter(user=phone))
        customer = Customer.objects.filter(phone=phone)
        for c in customer:
            name = c.name

        return render(request, 'search.html')  # koi product data pass hi nahi ho raha!
FIX 
from .models.product import Product

def search(request):
    query = request.GET.get('query')
    products = Product.objects.filter(name__icontains=query) if query else []

    context = {
        'products': products,
        'query': query,
    }
    return render(request, 'search.html', context)

और search.html में:

{% for product in products %}
    {{ product.name }}
{% endfor %}

यह Product table के name field में query text ढूंढेगा (icontains = case-insensitive partial match)।
